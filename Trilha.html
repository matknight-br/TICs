Trilha: Do Servidor Tradicional à Orquestração

1. Contextualização

O Ambiente Tradicional: Geralmente, você tem um servidor físico ou uma máquina virtual (ex: um servidor rodando Ubuntu). Você instala todas as dependências do sistema operacional manualmente, como pacotes, bancos de dados e linguagens de programação.

Principais Problemas:

  • Conflito de versões: Uma aplicação exige uma versão específica de uma biblioteca e outra aplicação no mesmo servidor exige uma versão diferente.
  • "Na minha máquina funciona": O ambiente de desenvolvimento é diferente do ambiente de produção.
  • Desperdício de recursos: Criar várias Máquinas Virtuais (VMs) consome muito processamento e memória (cada VM precisa de seu próprio Sistema Operacional).

O que é um ambiente orquestrado (Containerização): Em vez de virtualizar o hardware, nós empacotamos a aplicação e suas dependências em "Containers" (ex: Docker). O orquestrador (ex: Kubernetes ou Docker Swarm) é o maestro que gerencia esses containers: ele os reinicia se falharem, escala para mais máquinas se houver muito acesso e distribui a rede entre eles.

Vantagens: Isolamento total de processos, portabilidade, uso leve da infraestrutura e escalabilidade automatizada.

2. Imersão

Situação Problema

Você gerencia um servidor Ubuntu que hospeda um sistema legado em Python 2.7. Agora, a equipe de desenvolvimento criou uma nova API de inteligência artificial fantástica que exige estritamente o Python 3.10. Instalar as duas versões e suas bibliotecas diretamente no servidor está causando quebras no sistema antigo e conflitos de dependência no Ubuntu.

Análise da Infraestrutura

  • Ponto de falha atual: Acoplamento forte entre as aplicações e o SO hospedeiro.
  • Justificativa para orquestrar: Isolar a aplicação legado e a nova API em containers separados. Isso permite que ambas rodem na mesma máquina física, cada uma com seu próprio ambiente perfeitamente isolado.

3. Prática (Mão na Massa)

Vamos simular a migração desse ambiente virtualizado conflituoso para um orquestrado.

Ferramenta: Acesse o Replit ou use o terminal do seu próprio ambiente Linux para criar um projeto e escrever dois arquivos fundamentais.

  1. O Dockerfile (Empacotando a nova API): Crie a receita da sua aplicação definindo a imagem base, instalando dependências e expondo a porta.
  2. O docker-compose.yml (Mini-Orquestração): Escreva o arquivo que sobe o banco de dados e a aplicação simultaneamente, conectando-os em uma rede virtual isolada.

Desafio: Tente subir os dois containers com o comando docker-compose up -d e veja sua aplicação rodando isolada do sistema operacional host!

Flashcards de Revisão

Imagem Docker
Um pacote imutável contendo código, bibliotecas e configurações necessárias para rodar um container.
Dockerfile
Um arquivo de texto com as "receitas" (instruções) para construir uma Imagem Docker passo a passo.
Orquestrador
Ferramenta que automatiza a implantação, escalabilidade e gerenciamento de vários containers.
